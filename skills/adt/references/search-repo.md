# adt search_repo

Search Git commit history for database objects — by commit message, file name, object type, object name, author, date range, or commit number. Also supports restoring previous file versions from history.

ADT reads a pre-built YAML commit index file (generated by `adt patch -rebuild`) rather than parsing `git log` directly, so searches are fast even on large repositories.

**Prerequisite:** the commit index file must exist. If missing, run `adt patch -target {ENV} -rebuild` first.


## Common Usage


### Searching Commits

```bash
# Show all commits (subject to default commit range)
adt search_repo

# Search commits by words in the summary
adt search_repo -summary TASK-123
adt search_repo -summary refactor login

# Search for a specific file (even deleted/renamed files)
adt search_repo -file MY_PACKAGE

# Filter by object type (LIKE syntax)
adt search_repo -type PACKAGE%

# Filter by object name (LIKE syntax)
adt search_repo -name APP_CORE%

# Combine filters — find all view changes in the last 30 days
adt search_repo -type VIEW -recent 30

# Show only my commits
adt search_repo -my

# Limit to a specific commit range
adt search_repo -commits 10-50

# Search by specific commit hashes
adt search_repo -hash abc1234 def5678

# Limit to a specific branch
adt search_repo -branch feature/login

# Limit to recent days
adt search_repo -recent 14
```


### Restoring Previous Versions

```bash
# Restore all matched files to their historical versions (creates versioned copies)
adt search_repo -file MY_PACKAGE -restore

# Restore and stage as Git commits (overwrites original files)
adt search_repo -file MY_PACKAGE -restore -stage

# Full reset + restore + stage (resets to origin/DAILY first, then restores)
adt search_repo -file MY_PACKAGE -restore -stage -reset
```


## Flags

### Main Actions

| Flag | Purpose |
|---|---|
| `-file {NAME(S)}` | Search for commits containing files matching name(s) (case-insensitive substring match, space-separated) |
| `-name {PATTERN}` | Filter by object name (LIKE syntax with `%`, case-insensitive) |

### Scope & Filtering

| Flag | Purpose |
|---|---|
| `-type {PATTERN}` | Filter by object type (LIKE syntax, e.g. `PACKAGE%`, `VIEW`) |
| `-summary {WORD(S)}` | Search commit summaries for all provided words (AND logic — all words must match) |
| `-commits {RANGE}` | Limit to specific commit numbers (space-separated, supports ranges like `10-50` and open-ended like `5+`) |
| `-hash {HASH(ES)}` | Search for specific commit hashes (space-separated) |
| `-recent {DAYS}` | Only show commits from the last N days |
| `-branch {NAME}` | Use commits from a specific branch (defaults to active branch or config default) |
| `-my` | Show only your commits (matches Git user email) |

### Restore Actions

| Flag | Purpose |
|---|---|
| `-restore` | Restore matched files from their historical commit versions |
| `-stage` | Stage restored files as Git commits (overwrites originals instead of creating versioned copies) |
| `-reset` | Reset to `origin/DAILY` before restoring (use with `-stage`) |


## How It Works

1. Reads the pre-built commit index YAML file (path configured via `repo_commits_file` in config, with branch name substituted).
2. Iterates through commits, filtering by: commit range, date, author, summary words.
3. For each matching commit, filters its files by: object type, object name, file name.
4. Displays matching commits with their files grouped by object type.

Only database objects are shown — APEX files and other non-object files are filtered out (`obj.is_object` check). Deleted files are flagged with `[DELETED]`.


## Output Format

For each matching commit, the output shows:

```
{commit_num}) {commit_summary}
  {author}                                    {date}

    PACKAGE | APP_CORE
            | APP_UTILS
       VIEW | V_ACTIVE_USERS
      TABLE | CONFIG_PARAMETERS  [DELETED]
```

Files are grouped by object type, sorted in the standard ADT type order. Deleted files show a `[DELETED]` flag.


## Restore Behavior

When `-restore` is used:

- **Without `-stage`**: creates versioned copies next to the original file. For example, restoring `my_package.sql` from commit 15 creates `my_package.15.sql`. This lets you compare versions side by side without touching the current file.

- **With `-stage`**: overwrites the original file and creates a Git commit for each restored version. The commit message includes the original commit number and summary. Files are not pushed — they stay as local commits so you can explore changes and revert if needed.

- **With `-stage -reset`**: first runs `git fetch` + `git reset --hard origin/DAILY` to clean up any staged files and unpushed commits, then proceeds with the staged restore. Use this to start fresh when re-running restores.


## Tips

- If you get `COMMIT FILE MISSING`, run `adt patch -target {ENV} -rebuild` to generate the commit index.
- The `-summary` filter uses AND logic — all words must appear in the commit message. Use it with task/card numbers for precise matching.
- The `-file` filter is a case-insensitive substring match, not LIKE syntax. Passing `-file core` matches any file with "core" in its path.
- The `-name` and `-type` filters use LIKE syntax (with `%` wildcards), consistent with other ADT commands.
- Combine `-restore -stage` for a powerful "time travel" workflow: see how a file evolved commit by commit, with each version as a separate Git commit you can diff.
